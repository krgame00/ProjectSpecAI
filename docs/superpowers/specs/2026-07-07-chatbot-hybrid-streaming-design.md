# แผนการออกแบบ Chatbot Hybrid Streaming

## 1. ภาพรวม (Overview)
อัปเกรดแชทบอท `PCSpec` เดิม ให้รองรับการส่งข้อมูลแบบ Server-Sent Events (SSE) Streaming เพื่อให้ได้เอฟเฟกต์การพิมพ์ข้อความแบบเรียลไทม์ (ทีละคำแบบ ChatGPT) พร้อมนำฟีเจอร์การแสดงแหล่งอ้างอิงจาก Google Search (Grounding) และการรองรับรูปภาพ (Multimodal) จากโปรเจกต์ `ZCodeProject` เดิม มาผสานเข้ากับโครงสร้างปัจจุบัน

## 2. โครงสร้างสถาปัตยกรรม & กระแสข้อมูล (Architecture & Data Flow)

### 2.1 ระบบหลังบ้าน Backend (`routes/chatbot.js`)
- **สร้าง API Endpoint ใหม่:** `POST /api/chatbot/stream`
- **การประมวลผลสตรีม:** เราจะใช้คำสั่ง `ai.models.generateContentStream()`
- **กลยุทธ์การแยก JSON ออกจากข้อความ (JSON Separation Strategy):** เพื่อป้องกันไม่ให้โค้ด JSON ดิบๆ ไหลออกไปแสดงบนหน้าจอแชทของลูกค้า เราจะปรับปรุง `SYSTEM_INSTRUCTION` (คำสั่งสมองของ AI) ดังนี้:
  - สั่งให้ AI ตอบข้อความพูดคุยทั่วไปออกมาก่อน
  - จากนั้นให้ขั้นด้วยสัญลักษณ์พิเศษ เช่น `---JSON_START---`
  - ตามด้วยข้อมูลก้อน JSON (`{ "recommended_build": {...} }`)
- **การจัดการข้อมูลที่ถูกสตรีมมา (Chunk Handling):** 
  - เมื่อข้อมูลไหลเข้ามาทีละส่วน (Chunk) เซิร์ฟเวอร์จะคอยเช็คหาสัญลักษณ์พิเศษ
  - ข้อความที่อยู่ *ก่อนหน้า* สัญลักษณ์พิเศษ จะถูกสตรีมยิงตรงไปยังหน้าเว็บทันทีผ่าน `event: text` (เพื่อให้ลูกค้าเห็นข้อความพิมพ์ทีละคำ)
  - ข้อความที่อยู่ *หลังจาก* สัญลักษณ์พิเศษ จะถูกแอบเก็บสะสมไว้ในหน่วยความจำของเซิร์ฟเวอร์
  - เมื่อสตรีมจบ เซิร์ฟเวอร์จะนำข้อความที่สะสมไว้มาแปลงเป็น JSON และส่งไปที่หน้าเว็บในรวดเดียวผ่าน `event: build_data`
- **แหล่งอ้างอิง (Grounding Sources):** ดึงข้อมูลมาจาก `chunk.candidates[0].groundingMetadata` และส่งไปยังหน้าเว็บผ่าน `event: sources`
- **ประวัติแชท (History):** เราจะสร้างตัวแปร `sessions` แบบ Map เก็บไว้ในหน่วยความจำเซิร์ฟเวอร์ (เหมือนโปรเจกต์ ZCode) โดยอิงตาม `sessionId` เพื่อให้บอทจำบริบทการคุยได้
- **รูปภาพ (Images):** API จะรองรับการรับรูปภาพแบบ base64 และส่งต่อไปให้ Gemini วิเคราะห์ผ่าน `inlineData`

### 2.2 ระบบหน้าบ้าน Frontend (`ChatbotWindow.vue`)
- **ตัวรับข้อมูลสตรีม (SSE Consumer):** เปลี่ยนจากการใช้ `axios.post` ไปเป็นการใช้ `fetch` API ดั้งเดิม พร้อมกับ `response.body.getReader()` เพื่อแยกแยะข้อมูลสตรีม SSE
- **การจัดการ Event:**
  - `event: text` -> นำไปต่อท้ายตัวแปร `text` ของข้อความปัจจุบัน (ทำให้เกิดเอฟเฟกต์ตัวหนังสือวิ่ง)
  - `event: build_data` -> นำไปใส่ในตัวแปร `buildData` ของข้อความปัจจุบัน (เพื่อสั่งแสดงปุ่ม "หยิบลงตะกร้า" ในกล่องแชท)
  - `event: sources` -> นำไปใส่ในอาร์เรย์ `sources` (เพื่อแสดงป้ายชิปลิงก์อ้างอิงให้กดคลิกได้ใต้ข้อความ)
- **ปุ่มอัปโหลดรูปภาพ (Image Upload UI):** เพิ่มไอคอนคลิปหนีบกระดาษหรือรูปภาพตรงช่องพิมพ์แชท เมื่อลูกค้าเลือกรูป ระบบจะแปลงเป็น base64 และส่งไปพร้อมกับข้อความ
- **การจัดการ Session:** สร้างรหัสแบบสุ่ม UUID ขึ้นมา 1 ตัวตอนเปิดหน้าเว็บครั้งแรก เพื่อใช้เป็น `sessionId` ประจำแท็บนั้นๆ

## 3. การจัดการข้อผิดพลาด (Error Handling)
- หาก AI สร้าง JSON ออกมาผิดพลาด (หลังเครื่องหมายแบ่งส่วน) เซิร์ฟเวอร์จะตั้งค่า `build_data` เป็น `null` อย่างปลอดภัยเพื่อไม่ให้เว็บพัง
- หากการสตรีมถูกตัดกลางคัน หน้า UI จะทำการปิดกล่องข้อความและขึ้นไอคอนเตือน (Warning) ให้ทราบ

## 4. ข้อควรระวังและข้อแลกเปลี่ยน (Trade-offs & Considerations)
- **ข้อจำกัดหน่วยความจำ:** การจำประวัติแชทไว้ในหน่วยความจำของเซิร์ฟเวอร์ หมายความว่าประวัติแชทจะหายไปถ้าเรารีสตาร์ทเซิร์ฟเวอร์ (หากสเกลระบบใหญ่ขึ้น ค่อยเปลี่ยนไปใช้ Redis แทน)
- **การกิน Token:** การส่งรูปภาพจะกิน Token ค่อนข้างสูง แต่เนื่องจากเราต่อผ่าน Headroom Proxy ไว้แล้ว น่าจะช่วยลดความซ้ำซ้อนและประหยัด Token ไปได้ระดับหนึ่ง
