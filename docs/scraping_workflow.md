# 🕸️ ระบบ Data Pipeline อัตโนมัติ (Autonomous Scraping Workflow)

เอกสารนี้อธิบายสถาปัตยกรรมและขั้นตอนการทำงานของระบบดึงข้อมูลสินค้า (Hardware) จากเว็บไซต์ภายนอก (เช่น iHAVECPU) เข้าสู่ฐานข้อมูลของ ForgeLabs แบบอัตโนมัติ โดยใช้แนวคิด **Multi-Agent System** และ **Loop-Engineering**

---

## 🛠️ โครงสร้างของระบบ (Agents & Components)

ระบบนี้ถูกแบ่งการทำงานออกเป็น 4 ส่วนหลัก (เปรียบเสมือนพนักงาน 4 แผนก):

### 1. Agent 4: ฝ่ายจัดหา (Crawler / Spider)
* **หน้าที่:** เป็น "แมงมุม" เข้าไปที่หน้าหมวดหมู่สินค้าหลัก (เช่น หน้าจัดสเปค หรือหน้ารวม CPU) เพื่อกวาดตาดึงเอา "ลิงก์สินค้าแต่ละชิ้น" ทั้งหมดออกมา
* **ผลลัพธ์:** ได้รายชื่อ URL ของสินค้าทั้งหมด นำไปบันทึกเก็บไว้ในไฟล์ `urls.txt` แบบบรรทัดต่อบรรทัด
* **ระบบกรองข้อมูลซ้ำ (Deduplication):** ก่อนบันทึก Agent 4 จะตรวจสอบไฟล์ `scraped_urls.txt` (ประวัติการดึง) หากพบว่าลิงก์ไหนเคยดึงไปแล้ว จะถูกคัดทิ้งทันที เพื่อให้ได้เฉพาะข้อมูลใหม่เสมอ
* *(Note: ส่วนนี้สามารถต่อยอดใช้ Scrapling Spider ในการดึงลิงก์แบบขนานได้)*

### 2. Agent 2: ฝ่ายลงพื้นที่ (Scraper)
* **หน้าที่:** รับช่วงต่อจากไฟล์ `urls.txt` โดยจะเดินสายเข้าไปตามลิงก์ทีละเว็บ และใช้เครื่องมือ **Scrapling** (ข้ามระบบป้องกันบอทเช่น Cloudflare) เพื่อดึง HTML ออกมา
* **การแปลงข้อมูล:** ดึงข้อมูลสำคัญ เช่น ชื่อสินค้า, ราคา, รูปภาพ และสเปคเชิงลึก ออกมาจัดรูปแบบ (Data Mapping) ให้อยู่ในโครงสร้างมาตรฐานของระบบเรา
* **ไฟล์ที่เกี่ยวข้อง:** `agent2_scraper.py`

### 3. Agent 3: ฝ่ายตรวจสอบคุณภาพ (QC / Loop-Verifier)
* **หน้าที่:** ตรวจสอบความสมบูรณ์ของข้อมูลที่ Agent 2 แกะมาได้ (JSON) ก่อนที่จะส่งเข้าระบบจริง
* **กฎการตรวจสอบ (ตัวอย่าง):**
  - ราคาต้องไม่เป็น 0 หรือ NULL
  - ชื่อสินค้าและแบรนด์ต้องไม่ว่างเปล่า
  - ตารางสเปคต้องดึงมาได้ครบถ้วน
* **Loop-Budget (ฝ่ายบัญชี):** ถ้าระบบ QC ตรวจพบข้อผิดพลาด จะสั่งให้ Agent 2 กลับไปดึงข้อมูลใหม่ (ตีกลับ) แต่จะจำกัดโควต้าให้พยายามซ้ำได้สูงสุดแค่ 3 ครั้งต่อลิงก์ เพื่อป้องกันระบบบอทติดลูปและเปลืองทรัพยากร
* **ผลลัพธ์:** หากผ่าน QC ข้อมูลจะถูกรวบรวมแล้วเซฟเป็นไฟล์ `scraped_data.json`

### 4. Agent 1: ฝ่ายคลังสินค้า (Writer)
* **หน้าที่:** ทำหน้าที่นำข้อมูลที่การันตีความถูกต้องแล้วจาก `scraped_data.json` ยิงตรงเข้าสู่ฐานข้อมูล MySQL (`smart_pc_builder`) ของเว็บไซต์
* **จุดเด่น:** ยิงผ่านโค้ดสคริปต์ Node.js แบบตรงๆ ทำให้ทำงานได้รวดเร็วและไม่ต้องมานั่งขยับเมาส์บนหน้าเว็บ UI Admin
* **ระบบอัปเดตประวัติ:** ทันทีที่ยิงข้อมูลสินค้าชิ้นไหนสำเร็จ Agent 1 จะเอา URL ของสินค้านั้นไปต่อท้ายในไฟล์ `scraped_urls.txt` เพื่อเป็นฐานข้อมูลให้ Agent 4 ใช้กรองในรอบถัดไป
* **ไฟล์ที่เกี่ยวข้อง:** `node-backend/import_hardware.js`

### 5. Agent 5: ฝ่ายตรวจสอบหลังการขาย (Auditor / Post-Import QC)
* **หน้าที่:** ตรวจสอบความถูกต้องของข้อมูล **บนเว็บไซต์ของเรา** (UI / API) หลังจากที่ Agent 1 นำเข้า Database เรียบร้อยแล้ว เพื่อให้แน่ใจว่าหน้าตาเว็บแสดงผลไม่เพี้ยน
* **สิ่งที่ตรวจ:** เช็คว่ารูปภาพโหลดขึ้นจริง (Broken Links), ราคาบนหน้าเว็บตรงกับต้นฉบับ, และสเปค (Socket, TDP) แสดงผลถูกต้องในหน้าเลือกสินค้า
* **ผลลัพธ์:** สร้างรายงาน (Audit Report) หรือแจ้งเตือนทันทีหากพบข้อมูลที่ผิดปกติโผล่บนหน้าเว็บลูกค้า

## 🚀 ลำดับขั้นตอนการสั่งงาน (Execution Flow)

เมื่อต้องการอัปเดตหรือเพิ่มสินค้าล็อตใหม่เข้าสู่ระบบ ให้รันสคริปต์ตามลำดับต่อไปนี้:

1. **หารายชื่อสินค้า (รัน Agent 4)**
   *(ในกรณีที่ยังไม่มีสคริปต์ Agent 4 อัตโนมัติ สามารถก๊อปปี้ URL มาแปะใน `urls.txt` เองได้)*
   ```bash
   # Input: urls.txt
   ```

2. **ดึงข้อมูลและตรวจ QC (รัน Agent 2 + Agent 3)**
   คำสั่งรันสคริปต์ Python เพื่อเปิดระบบ Scraper ควบคู่กับการทำ QC
   ```bash
   python agent2_scraper.py
   # Output: scraped_data.json
   ```

3. **นำเข้าฐานข้อมูล (รัน Agent 1)**
   คำสั่งรันสคริปต์ Node.js เพื่อบันทึกข้อมูลเข้า Database
   ```bash
   node node-backend/import_hardware.js
   # Output: สินค้าอัปเดตขึ้นเว็บ ForgeLabs ทันที
   ```

4. **ตรวจเช็คหน้าเว็บจริง (รัน Agent 5)**
   (ระบบ Automation Test เพื่อเช็คหน้าเว็บหลังบ้าน/หน้าบ้าน)
   ```bash
   # (รอการพัฒนาสคริปต์ Agent 5)
   # Output: Audit Report (Pass / Fail)
   ```

## 📌 เทคโนโลยีที่ใช้ (Tech Stack)
- **Python (Scrapling):** สำหรับทำหน้าที่ Scraper ฝ่าระบบป้องกันบอท (Cloudflare)
- **Node.js + MySQL2:** สำหรับเชื่อมต่อและจัดการฐานข้อมูล
- **JSON:** รูปแบบไฟล์กลางที่ใช้รับส่งข้อมูลระหว่างกระบวนการ
